My stack will outlive yours

December 2020

My stack requires no maintenance, has perfect Lighthouse scores, will never have any security vulnerability, is based on open standards, is portable to any host, has an instant dev loop, has no build step and… will outlive any other stack.

It’s not LAMP, Wordpress, Rails, MEAN, Jamstack. I don’t do CSR (Client-side rendering), SSR (Server Side Rendering), SSG (Static Site Generation)...

My stack is HTML+CSS.

And because my sources are in git, pushed to GitHub, GitHub Pages is my host.

Of course, I’m being a bit provocative here. I should rather say that I use only HTML+CSS for some specific use cases where others might be relying on some more popular technologies. Because I’m done rewriting my site every couple of years.

Why HTML+CSS?

It all started with a blog, that I was hosting on Wordpress.com (which is "Wordpress-as-a-Service", because the last thing I want to do is be the administrator of a custom Wordpress installation). I paid Wordpress.com to do one job: host my blog. And one day I looked at its sources and Lighthouse scores:

What have we done?

Sure, this is just one specific blogging platform and I’m pretty sure others are better, at least in terms of performance. But isn’t there something fundamentally wrong if displaying a short text article with some images takes seconds, loads countless render-blocking scripts, and has unreadable page sources?

My requirements were simplicity and performance.

It was time to say goodbye to Wordpress. I didn’t need 99% of its features anyway. Other blogging platforms didn’t meet expectations either (I seriously have no idea why so many people publish on Medium… behind a login wall). I looked at static site generators like Jekyll, Hugo, 11ty, but all of these require tooling installed, have a build step and will ultimately need some updates. What if we also get rid of these?

What is the HTML+CSS stack good for?

Let’s first differentiate between a web page and a web app: a web page’s goal is to serve content, on the other hand, the web app’s goal is to enable you to perform interactive tasks. There are in-betweens too, often in the form of content that might need customization depending on the logged in user and content that might allow some interactions.

HTML+CSS fits the web page use case. Wow, what a revelation! It might seem obvious, but it seems we’ve all forgotten this these days. HTML+CSS does not fit the web app use case, or any in between.

We said a web page serves content, but let’s dive into more concrete use cases:

  • Product / Company / Business landing page and marketing sites
  • Personal portfolio / bio
  • Blog
  • Documentation

How to develop for HTML+CSS?

Authoring a pure HTML+CSS site can be done in any text editor, in any environment (any desktop OS, any smartphone, or even directly using GitHub’s single file editor) and previewed by simply opening the file in any browser.

Keep the HTML of every page minimal and semantic. First, because there is no need for countless of <div>, but then because it makes sources more readable and easier to edit. Creating a new page requires you to clone an existing one.

When not cluttered with unnecessary scripts, divs or classes, I have no issue writing HTML directly. Yes, the paragraph tags are a bit annoying and distracting, but proper indentation and syntax highlighting mitigate this.

Sometimes, for drafting long blog articles, I’m working in Google Docs, and when I’m happy, export the content to clean HTML using an add-on. Google Docs is awesome for collaboration, with a powerful suggestions and commenting system. That’s ideal for the “draft” phase.

Because all content is in git, approval and doing a final review can be done via GitHub pull requests.

When I publish a new page, I need to link to it manually from the index page. I’m OK with that. It’s done in one line. It also allows me to have more control over when I want the page to be “published”.

But... if I don’t use any templating system, how do I update my header, footer or nav? Well, simply by using the ”Replace in files” feature of any good text editor. They don’t need frequent updates anyway. The benefits of using a templating system is not worth the cost of introducing it.

In conclusion

You don’t need React, Next.js, Wordpress, or Hugo to put a web page or blog online. Raw HTML and CSS do the job.

That being said, you’ll need to pick up some tooling or framework if you want to build a web app or add some interactivity or customization to your web pages. And for that, I’m very glad to see that frameworks now seem to be prioritizing performance, notably by serving the HTML over the wire and leveraging caching whenever possible. The era of “download 5MB of JS first and then download content from an API” seems to be over for content web pages. (I personally think it’s still OK to do so for web apps,as the user’s intent is different).

Is the “HTML+CSS only” approach a bit extreme? A bit, it’s basically saying “all software is terrible, how can minimize depending on software at all”. Web standards are a model of backward compatibility. I’m pretty confident that my web pages written in raw HTML+CSS will have no issue being accessed 10 years from now, and this without me having to do anything.
